# Python Stable Rules

## A. 修改前要求
1. 先识别任务范式：IO / CPU / 流 / 领域 / 脚本 / 异步 / 调度。
2. 先分析现有模块的实现风格、数据流和边界对象。
3. 优先复用现有 service / repository / schema / util / config 模式。

## B. 设计规则
1. 公共函数、跨模块函数必须有明确 Type Hints。
2. 无理由禁止 Any。
3. 外部输入边界必须显式校验。
4. 跨层或跨边界数据优先使用显式 schema（优先 Pydantic）。
5. 禁止擅自改变现有接口契约。
6. 未经证明，不新增抽象层。

## C. 稳健性规则
1. 入口优先 Guard Clauses。
2. 禁止裸 except。
3. 禁止无说明的 except Exception。
4. 文件、网络、锁、数据库连接必须显式关闭或使用上下文管理。
5. 配置禁止硬编码，统一从配置对象或环境变量读取。

## D. 性能与范式规则
1. IO 密集任务优先考虑异步。
2. CPU 密集任务优先考虑多进程或进程池。
3. 大数据处理优先流式/迭代式设计，避免一次性载入。
4. 未经测量，不因“理论更快”而牺牲可读性。
5. 禁止在高频循环里直接发起 DB / 网络请求，除非已有批处理策略或明确说明原因。

## E. 文档与可验证性
1. 模块、复杂函数、公共接口必须有文档说明。
2. 复杂度标注只要求在性能敏感或非平凡函数上出现。
3. 核心逻辑必须可验证：
   - 优先 pytest
   - 小型脚本可用 __main__ 最小验证

## F. 结构性设计规则

### F1. 领域建模
1. 对核心领域对象，优先把关键业务不变量、状态流转和约束收敛到模型内部。
2. DTO / Schema 主要用于边界契约，不应替代全部领域建模。
3. 当复杂领域规则长期散落在多个 service / handler 中时，应视为贫血模型风险。
4. 纯 CRUD、无复杂业务规则场景，可接受 Transaction Script / 简单数据模型，不必强行充血建模。

### F2. 总线使用
1. 总线只用于跨模块、跨边界或一对多通知场景。
2. 如果只是单一同步调用链，优先直接函数调用或服务调用。
3. 禁止为了“看起来解耦”而把本地简单调用改造成总线事件。
4. 对非关键通知型副作用事件，可对单个 handler 做异常隔离；对关键业务事件，不应默认吞异常继续。

### F3. 注册机制
1. 只有在真实扩展点存在时才引入 registry。
2. registry 必须具备：
   - 唯一 key
   - 重复注册检测
   - 查找失败时报错
   - 可列举已注册项（推荐）
3. 禁止依赖裸全局 dict + 导入副作用完成关键注册。
4. 有容器隔离和显式发现机制的注册方式可以接受；仅靠 import 碰运气的隐式注册属于高风险设计。

### F4. 分发机制
1. 分发标准必须单一、显式、可测试。
2. 类型集合开放、分支持续增长时，优先 singledispatch、表驱动或注册式分发。
3. 类型集合封闭、分支少且逻辑简单时，if/elif 可接受。
4. 路由或规则分发中，更具体规则必须优先于更一般规则。
5. 针对 Starlette / FastAPI 这类按注册顺序匹配的框架，固定路径应优先于参数路径。

### F5. 抽象层
1. 抽象层必须服务于真实替换需求、测试隔离或稳定边界。
2. 无真实多实现、无测试替换需要、无对外扩展场景时，不应新增抽象层。
3. 抽象接口不得泄漏具体实现专属概念。
4. 在 ABC + @abstractmethod 场景下，方法体可用 `...` 保持简洁；`raise NotImplementedError` 不是错误，只是通常没有必要再写一遍。

### F6. 组合式管线
1. 管线中的每一步必须输入输出契约清晰。
2. 步骤应尽量无隐藏共享状态。
3. 每一步应可单独测试、可独立复用。
4. 管线顺序必须显式。
5. 禁止把大量逻辑塞进一个 mega-step 却称其为“管线”。
