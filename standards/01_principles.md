# 开发规范总原则

## 核心目标
规范的目标不是增加形式主义，而是：
1. 提高可维护性
2. 提高稳健性
3. 提高扩展性
4. 保持行为稳定
5. 降低 AI 与人工协作时的歧义

## 决策顺序
所有编码与修改任务必须按以下顺序思考：

1. 场景分类
- IO 密集
- CPU 密集
- 数据流处理
- 核心领域逻辑
- 胶水/脚本任务
- 并发/异步/任务调度
- 元编程/动态行为

2. 非功能属性评估
- 可维护性
- 稳健性
- 扩展性
- 原子性
- 可测试性
- 性能敏感度

3. 考古式分析
- 先理解现有实现
- 先找已有模式、已有封装、已有调用链
- 优先优化现有结构，而不是重写

4. 范式决策
- 只有在能说明理由时，才切换范式或引入更复杂结构
- 不因“理论更高级”而替换现有简单实现

5. 最小改动落地
- 优先局部修改
- 优先复用现有抽象
- 优先兼容已有接口与行为

## 规则准入标准
任何新规则要进入 stable，必须同时满足：
1. 在真实代码/评审/AI误改中重复出现至少 2 次
2. 不遵守会导致可量化风险
3. 10 秒内可理解
4. 不与现有 linter / formatter / type checker 重复
5. 适用范围清晰
6. 能说明例外场景

## 规则废止标准
以下规则应降级或删除：
1. 长期不触发
2. 成本明显高于收益
3. 已被自动化工具覆盖
4. 与当前项目架构不再匹配
5. 导致 AI 或人工频繁误伤正常开发
