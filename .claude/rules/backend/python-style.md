# Python 后端开发规范

## 总体原则
- 先判断任务场景，再选择合适范式：
  - IO 密集：优先异步
  - CPU 密集：优先多进程或并行方案
  - 大数据流：优先生成器/迭代器
  - 核心业务：优先类型化建模
  - 简单胶水代码：优先小而清晰的函数式组织
- 不要为了“高级”而过度设计；以清晰、稳定、可维护为第一目标。

## 类型与契约
- 所有公开函数、方法、类接口都必须写明确的 Type Hints。
- 无充分理由不要使用 `Any`。
- 跨函数、跨层、跨接口传递的结构化数据，优先使用明确的数据模型；如项目已采用 Pydantic，则继续沿用，避免裸 Dict 到处传递。
- 输入输出契约必须清晰，字段命名保持稳定。

## 健壮性
- 入口处优先做边界检查和 guard clauses。
- 禁止裸 `except:`。
- 尽量捕获具体异常，并提供清晰日志或错误上下文。
- 错误处理不要吞异常；必要时记录并重新抛出。

## 资源安全
- 文件、网络连接、数据库连接、锁、临时资源，优先使用上下文管理。
- 不要遗留未关闭资源。

## 配置管理
- 禁止硬编码敏感信息、地址、密钥、账号和环境差异配置。
- 配置统一从环境变量、配置类或 settings 模块读取。

## 文档要求
- 新文件或复杂模块应写模块头注释，说明：
  - 模块用途
  - 外部依赖/配置
  - 关键数据流
  - 主要函数或类
- 公开函数和核心逻辑建议写 docstring。
- 对非平凡逻辑，说明时间复杂度和空间复杂度。

## 工程实践
- 优先写小函数、单一职责函数。
- 优先复用已有工具层、服务层、数据模型和异常模型。
- 不做无必要的层级增加和抽象堆叠。
- 默认补充针对性的 pytest 测试；仅对脚本型文件再考虑加入 `if __name__ == "__main__":` 演示入口。
- 改动后至少运行最小必要验证，例如 `ruff`、`pytest`、`mypy` 中相关部分。

## AI 执行要求
- 编写或修改代码前，先判断本次任务属于 IO / CPU / 流式 / 领域 / 脚本中的哪一类，并说明原因。
- 先给出实现计划，再开始写代码。
- 修改后总结：
  - 采用了什么范式
  - 契约模型是什么
  - 做了哪些边界检查
  - 捕获了哪些具体异常
  - 运行了哪些验证命令
