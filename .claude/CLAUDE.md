# 项目级协作说明

## 项目概况
- 项目名称：FormFlow
- 项目类型：高校多租户表单与审批中台
- 架构：Vue3 前端 + Python FastAPI 后端
- 后端技术栈：Python 3.10 / FastAPI / SQLAlchemy / Alembic / PostgreSQL / Redis / MinIO / APScheduler / JWT
- 前端技术栈：Vue 3 / Vite / TypeScript / Naive UI / Pinia / Vue Router / Axios

## 全局工作方式
- 先理解需求与现有实现，再给出计划，再实施，再验证。
- 对非小改动，先说明会影响哪些文件、模块、接口、页面或数据结构。
- 如果需求不明确，先提问澄清，不要直接做大改。
- 优先遵循现有架构、命名方式、目录边界和代码风格，不随意引入新的组织方式。
- 只修改与当前任务直接相关的文件，避免顺手重构无关代码。

## 代码修改原则
- **考古式分析**：在修改任何代码之前，必须先分析现有代码的实现逻辑、依赖关系、数据流向和设计模式。工作目标是优化、修复或增量演进，而不是无边界重写。
- **最小化改动**：每次修改应优先局部调整并复用现有结构；仅当现有结构明显不利于维护、复用或测试时，才新增函数、类或文件。
- **禁止功能漂移**：不得以“优化”为名改变原有功能的行为、输入输出、返回结构、副作用或异常语义。若发现潜在问题，应先说明并等待确认。
- **遵循项目规范**：严格遵守现有命名规范、目录结构、Pinia 状态管理方式、Vue Router 路由组织方式、API 调用模式和后端分层方式。
- **破坏性改动先确认**：如果修改会影响接口字段、数据库结构、公共函数签名、Pydantic 模型、前后端契约、租户逻辑或鉴权流程，必须先说明影响范围、兼容性风险和迁移方案，再等待确认。
- **输出前先验证**：完成关键改动后，必须优先运行最小必要验证，并明确说明运行了哪些命令、结果如何、还剩哪些未覆盖风险。

## 重构与兼容性原则
- 默认保持可观察行为不变，除非任务明确要求改变行为。
- 优先保持向后兼容。
- 重构应按小步、可验证、可回退的方式进行，不做无边界大改写。
- 避免无必要的大规模重命名、目录搬迁、接口签名变化和状态结构重组。
- 如果必须引入破坏性改动，先停止实施，先说明影响范围、调用方、迁移方案和风险。

## 项目结构
### 后端关键目录
- `app/api/v1/`: API 路由层
- `app/services/`: 业务服务层
- `app/models/`: ORM 模型
- `app/schemas/`: Pydantic 数据模型
- `app/core/`: 基础设施（数据库、Redis、安全等）
- `app/middleware/`: 中间件
- `alembic/`: 数据库迁移脚本

### 前端关键目录
- `src/api/`: API 接口定义
- `src/components/`: 可复用组件
- `src/views/`: 页面组件
- `src/stores/`: Pinia 状态管理
- `src/types/`: TypeScript 类型定义
- `src/utils/`: 工具函数
- `src/router/`: 路由配置

## 后端开发约定
- 路由层只负责输入输出、鉴权、权限校验和调用服务；业务逻辑下沉到 `services`。
- FastAPI 接口优先使用 `async def`，并补齐类型标注。
- 数据库依赖统一来自 `get_db`，保持清晰的请求级事务边界。
- 跨层数据传递优先使用明确的数据模型，延续现有 Pydantic 约定。
- 配置统一从 `settings` 或 `.env` 读取，禁止硬编码密钥、地址和环境差异配置。
- 外部依赖（Redis、MinIO、AI 接口等）必须封装到可替换的服务层。
- 保持统一响应和异常处理方式，遵循现有 `success_response` / `error_response` 和异常体系。
- 日志应延续结构化日志和 `request_id` 传递方式。

## 前端开发约定
- 组件默认使用 `<script setup lang="ts">`。
- 优先使用 Composition API，保持类型安全。
- 跨组件共享状态优先通过 Pinia 管理，不在组件之间堆积隐式副作用。
- 路由新增保持懒加载，并在 `src/router/` 统一管理。
- API 请求统一通过 `src/api/` 封装，不在页面组件里散落重复请求逻辑。
- 与后端响应结构强相关的字段、类型和错误处理逻辑要集中维护，避免多处手写复制。
- 优先复用已有组件、composables、stores、types 和 utils。

## 前后端联动规则
- 涉及接口字段、状态码、分页、过滤、排序、认证、租户、文件上传、导出下载的改动，必须同时检查前后端影响。
- 后端返回结构变化时，同步检查前端 API 层、类型定义、页面展示和错误处理。
- 前端表单字段变化时，同步检查后端 schema、默认值、校验和兼容性。
- 涉及 JWT、租户头 `X-Tenant-ID`、权限控制时，默认按高风险改动处理。

## API 约定
- 后端 API 前缀：`/api/v1`
- 认证方式：Bearer Token (JWT)
- 多租户请求头：`X-Tenant-ID`
- 后端响应格式：统一使用现有响应封装
- 前端开发环境通过 Vite 代理 `/api` 到后端
- 所有 API 调用都应维护对应的 TypeScript 类型或接口定义

## 常用命令
### 后端
- 安装依赖：`pip install -r requirements.txt`
- 启动开发服务器：`uvicorn app.main:app --reload --port 8000`
- 数据库迁移：`alembic revision --autogenerate -m "描述"` / `alembic upgrade head`
- 测试：`pytest`
- 代码检查：`ruff check .`
- 格式化：`black .`
- 类型检查：`mypy .`

### 前端
- 安装依赖：`npm install`
- 启动开发服务器：`npm run dev`
- 构建：`npm run build`
- 预览：`npm run preview`
- 类型检查：`npm run type-check`
- 代码检查：`npm run lint`

## 验证要求
- 后端改动后，优先运行最小必要验证：`pytest`、`ruff check`、`mypy` 中相关部分。
- 前端改动后，优先运行最小必要验证：`npm run lint`、`npm run type-check`、`npm run build` 中相关部分。
- 涉及接口联动时，至少说明前后端两侧分别验证了什么。
- 如果无法运行验证，必须明确说明未运行项和原因，不得默认“应该没问题”。

## 输出要求
- 修改前：先给简短计划。
- 修改后：必须总结
  - 改了哪些文件
  - 每个文件改了什么
  - 是否有兼容性影响
  - 运行了哪些验证
  - 还存在哪些风险或后续建议
