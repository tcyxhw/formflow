# 审批流程配置器 UX 缺陷修复设计

## 概述

审批流程配置页面（/flow/configurator）存在三个关键缺陷，影响用户配置审批流程的能力。这个设计文档规范化了缺陷条件和验证方法，确保修复是有针对性的、最小化的，并且不会引入回归。

## 术语表

- **Bug_Condition (C)**: 触发缺陷的条件 - 用户在配置审批流程时的特定操作
- **Property (P)**: 期望的正确行为 - 系统应该如何响应这些操作
- **Preservation**: 必须保持不变的现有行为 - 其他功能不应受到修复的影响
- **FlowNodeInspector**: 位于 `my-app/src/components/flow-configurator/FlowNodeInspector.vue` 的节点检查器组件，负责显示和编辑节点属性
- **FlowRouteEditor**: 位于 `my-app/src/components/flow-designer/FlowRouteEditor.vue` 的路由编辑器组件，负责编辑路由属性
- **flowDraft store**: 位于 `my-app/src/stores/flowDraft.ts` 的 Pinia 状态管理，管理流程的节点和路由数据
- **selectedNodeKey**: 当前选中的节点的唯一标识符
- **selectedRouteIndex**: 当前选中的路由在路由数组中的索引

## 缺陷详情

### 缺陷条件

审批流程配置页面存在三个独立的缺陷，分别影响不同的功能：

**缺陷 1：条件设置交互问题**

当用户在条件节点上点击"编辑条件"按钮时，系统应该打开一个模态框进行条件编辑，但当前实现可能存在问题。

**缺陷 2：开始/结束节点显示问题**

当用户选择开始节点或结束节点时，系统在右下角显示条件设置和路由属性配置选项，但这些节点不需要这些配置。

**缺陷 3：路由属性覆盖问题**

当用户为多个节点设置路由属性时，后设置的路由会覆盖之前设置的路由。例如：
- 用户为审批节点设置路由：开始节点 → 审批节点
- 用户为结束节点设置路由：审批节点 → 结束节点
- 结果：第一条路由被删除，只保留第二条路由

**正式规范：**

```
FUNCTION isBugCondition(action)
  INPUT: action of type UserAction
  OUTPUT: boolean
  
  RETURN (action.type = 'EDIT_CONDITION' AND action.nodeType = 'condition' AND NOT modalOpened)
         OR (action.type = 'SELECT_NODE' AND action.nodeType IN ['start', 'end'] AND configOptionsShown)
         OR (action.type = 'UPDATE_ROUTE' AND previousRouteExists AND previousRouteDeleted)
END FUNCTION
```

### 示例

**缺陷 1 示例**：
- 用户在条件节点上点击"编辑条件"按钮
- 期望：打开一个模态框进行条件编辑
- 实际：可能没有打开模态框或打开了错误的界面

**缺陷 2 示例**：
- 用户选择开始节点
- 期望：右下角只显示基本节点信息（名称、类型）
- 实际：显示了条件设置和路由属性配置选项

**缺陷 3 示例**：
- 用户设置路由：开始节点 → 审批节点（路由 1）
- 用户设置路由：审批节点 → 结束节点（路由 2）
- 期望：两条路由都被保存
- 实际：路由 1 被删除，只保留路由 2

**边界情况示例**：
- 用户在结束节点上进行操作，系统不应该显示任何配置选项
- 用户设置多条路由（3 条或更多），所有路由都应该被独立保存

## 期望行为

### 保留需求

**不变的行为：**
- 普通审批节点应该继续显示条件设置和路由属性配置选项
- 其他 UI 元素的交互应该继续正常工作
- 审批流程的保存和加载功能应该继续正常工作
- 节点的基本属性编辑（名称、类型等）应该继续正常工作

**范围：**
所有不涉及开始/结束节点配置和路由覆盖的操作都应该完全不受此修复的影响。这包括：
- 审批节点的配置
- 条件节点的配置
- 自动节点的配置
- 路由的创建和删除
- 流程的保存和发布

## 假设的根本原因

基于缺陷描述，最可能的问题是：

1. **条件设置交互问题**：
   - FlowNodeInspector 组件中的 `openConditionModal` 方法可能没有正确实现
   - 模态框的显示状态管理可能有问题
   - 条件编辑器的集成可能不完整

2. **开始/结束节点显示问题**：
   - FlowNodeInspector 组件中的 `shouldShowApprovalConfig` 和 `shouldShowConditionConfig` 方法可能没有正确排除开始/结束节点
   - 节点类型检查逻辑可能不完整
   - 条件配置部分的渲染条件可能不正确

3. **路由属性覆盖问题**：
   - flowDraft store 中的 `updateRoute` 方法可能在更新路由时意外删除了其他路由
   - 路由的唯一标识符可能没有正确处理
   - 路由数组的管理逻辑可能有缺陷

## 正确性属性

Property 1: 缺陷条件 - 条件设置、节点显示和路由管理

_对于任何_ 用户操作，其中缺陷条件成立（isBugCondition 返回 true），修复后的系统应该：
- 在条件节点上点击"编辑条件"时打开模态框
- 在开始/结束节点上不显示条件设置和路由属性配置选项
- 在设置多个路由时保留所有之前设置的路由

**验证需求：2.1, 2.2, 2.3, 2.4, 2.5**

Property 2: 保留 - 非缺陷条件行为

_对于任何_ 用户操作，其中缺陷条件不成立（isBugCondition 返回 false），修复后的系统应该产生与原始系统相同的结果，保留所有现有功能。

**验证需求：3.1, 3.2, 3.3, 3.4**

## 修复实现

### 所需更改

假设我们的根本原因分析是正确的：

**文件 1**: `my-app/src/components/flow-configurator/FlowNodeInspector.vue`

**修改 1：修复开始/结束节点显示问题**
- 在条件配置部分添加节点类型检查，排除开始/结束节点
- 确保 `shouldShowConditionConfig` 方法返回 false 对于开始/结束节点
- 验证条件配置部分的 `v-if` 指令正确检查节点类型

**修改 2：确保条件设置弹窗正确实现**
- 验证 `openConditionModal` 方法正确设置了 `showConditionModal` 状态
- 确保模态框的 `v-model:show` 绑定正确
- 验证条件编辑器组件正确传递了所有必要的 props

**文件 2**: `my-app/src/stores/flowDraft.ts`

**修改 3：修复路由覆盖问题**
- 检查 `updateRoute` 方法是否意外修改了其他路由
- 确保路由数组的更新只影响指定索引的路由
- 验证路由的唯一标识符处理逻辑

## 测试策略

### 验证方法

测试策略采用两阶段方法：首先在未修复代码上表现缺陷，然后验证修复有效并保留现有行为。

### 缺陷条件探索检查

**目标**：在实现修复之前表现缺陷，确认或反驳根本原因分析。

**测试计划**：编写测试来模拟用户操作，验证系统的响应。在未修复代码上运行这些测试以观察失败并理解根本原因。

**测试用例**：
1. **条件设置弹窗测试**：在条件节点上点击"编辑条件"按钮，验证模态框是否打开（在未修复代码上会失败）
2. **开始节点显示测试**：选择开始节点，验证右下角不显示条件设置和路由属性配置选项（在未修复代码上会失败）
3. **结束节点显示测试**：选择结束节点，验证右下角不显示条件设置和路由属性配置选项（在未修复代码上会失败）
4. **路由覆盖测试**：设置多条路由，验证所有路由都被保留（在未修复代码上会失败）

**期望的反例**：
- 条件设置弹窗没有打开或打开了错误的界面
- 开始/结束节点显示了不应该显示的配置选项
- 后设置的路由覆盖了之前设置的路由

### 修复检查

**目标**：验证对于所有缺陷条件成立的输入，修复后的函数产生期望的行为。

**伪代码**：
```
FOR ALL action WHERE isBugCondition(action) DO
  result := fixedSystem(action)
  ASSERT expectedBehavior(result)
END FOR
```

### 保留检查

**目标**：验证对于所有缺陷条件不成立的输入，修复后的函数产生与原始函数相同的结果。

**伪代码**：
```
FOR ALL action WHERE NOT isBugCondition(action) DO
  ASSERT originalSystem(action) = fixedSystem(action)
END FOR
```

**测试方法**：基于属性的测试推荐用于保留检查，因为：
- 它自动生成许多测试用例
- 它捕获手动测试可能遗漏的边界情况
- 它提供强有力的保证行为不变

**测试计划**：首先在未修复代码上观察非缺陷输入的行为，然后编写基于属性的测试来捕获该行为。

**测试用例**：
1. **审批节点配置保留**：验证审批节点的配置继续正常工作
2. **条件节点配置保留**：验证条件节点的配置继续正常工作
3. **路由创建保留**：验证路由的创建继续正常工作
4. **流程保存保留**：验证流程的保存继续正常工作
