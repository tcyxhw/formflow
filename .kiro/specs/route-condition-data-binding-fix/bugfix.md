# 路由和条件数据绑定修复

## 介绍

当前审批流配置系统存在严重的数据混乱问题。条件和路由属性没有正确关联到具体的路由，导致在编辑不同节点时显示错误的条件数据，以及路由属性被多个节点共享。本修复旨在确保每个路由有独立的条件和属性，不会相互影响。

## Bug 分析

### 当前行为（缺陷）

1.1 WHEN 用户在节点 A 添加条件后切换到节点 B THEN 系统仍然显示节点 A 的条件，即使节点 B 有不同的路由

1.2 WHEN 用户编辑节点 B 的路由属性 THEN 系统显示所有路由而不是只显示进入节点 B 的路由（to_node_key = B）

1.3 WHEN 用户在节点 A 修改路由优先级或默认状态 THEN 系统没有正确区分不同的路由，导致多个路由共享同一个属性值

1.4 WHEN 用户编辑条件构建器中的条件 THEN 系统没有明确指出这是哪个路由的条件，导致条件被错误地保存到其他路由

### 期望行为（正确）

2.1 WHEN 用户在节点 A 添加条件后切换到节点 B THEN 系统只显示与节点 B 相关的路由的条件，不显示节点 A 的条件

2.2 WHEN 用户编辑节点 B 的路由属性 THEN 系统只显示进入节点 B 的路由（to_node_key = B），不显示从节点 B 出发的路由

2.3 WHEN 用户在节点 A 修改路由优先级或默认状态 THEN 系统正确区分不同的路由，每个路由有独立的属性值

2.4 WHEN 用户编辑条件构建器中的条件 THEN 系统明确指出这是哪个路由的条件（例如"从开始节点到审批节点的条件"），并将条件保存到正确的路由

### 未改变的行为（防止回归）

3.1 WHEN 用户创建新的审批流 THEN 系统仍然能够正确创建节点和路由，不影响基本的流程创建功能

3.2 WHEN 用户保存审批流配置 THEN 系统仍然能够正确保存所有节点和路由数据到数据库

3.3 WHEN 用户查看已保存的审批流 THEN 系统仍然能够正确加载并显示所有节点和路由数据

3.4 WHEN 用户删除节点 THEN 系统仍然能够正确删除相关的路由和条件数据

3.5 WHEN 用户在条件构建器中添加条件规则 THEN 系统仍然能够正确验证条件表达式的有效性
