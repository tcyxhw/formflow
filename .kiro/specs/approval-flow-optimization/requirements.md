# 审批流程优化 - 需求文档

## 一、项目概述

### 1.1 项目背景

FormFlow 审批流程系统已有基础架构，但与设计文档相比存在关键功能缺失：
- ❌ 缺失 CONDITION 节点（条件分支）
- ❌ 缺失驳回策略（RejectStrategy）
- ❌ 缺失审批操作 API
- ❌ 条件表达式格式不统一
- ❌ 缺失流程设计器 UI

### 1.2 项目目标

在 5-6 周内完成审批流程系统的优化，使其与设计文档完全对应，支持完整的审批流程。

### 1.3 项目范围

**第一阶段（P0 优先级）**：核心功能实现
- 条件表达式格式统一
- CONDITION 节点实现
- 驳回策略实现
- 审批操作 API
- 表单字段 API

**第二阶段（P1 优先级）**：UI 和查询功能
- 流程设计器 UI
- 查询接口（待办、已办、我发起的、时间线）
- 条件评估器完善

---

## 二、功能需求

### 2.1 条件表达式格式统一

**需求描述**：
统一前后端条件表达式格式，使用设计文档中定义的条件树格式。

**功能点**：
1. 后端支持设计格式的条件树
2. 前端条件构造器保持不变
3. 提供转换层支持现有 JsonLogic 条件迁移

**验收标准**：
- ✅ 后端条件评估器支持设计格式
- ✅ 所有条件表达式正确评估
- ✅ 现有 JsonLogic 条件可正确转换
- ✅ 单元测试覆盖率 > 80%

---

### 2.2 CONDITION 节点实现

**需求描述**：
实现条件分支节点，支持根据条件表达式动态路由到不同的下一个节点。

**功能点**：
1. 数据库支持条件分支配置
2. 后端支持条件分支评估和路由
3. 前端支持条件分支编辑
4. 流程校验支持条件节点

**验收标准**：
- ✅ 条件节点可正确创建和配置
- ✅ 条件分支可正确评估
- ✅ 流程可根据条件正确路由
- ✅ 流程校验包含条件节点检查
- ✅ 集成测试通过

---

### 2.3 驳回策略实现

**需求描述**：
实现审批驳回策略，支持驳回到发起人或上一个审批节点。

**功能点**：
1. 数据库支持驳回策略配置
2. 后端支持两种驳回策略：TO_START 和 TO_PREVIOUS
3. 前端支持驳回策略选择
4. 驳回流程正确处理

**验收标准**：
- ✅ 驳回策略可正确配置
- ✅ TO_START 策略正确处理（流程结束）
- ✅ TO_PREVIOUS 策略正确处理（重新审批）
- ✅ 单元测试通过

---

### 2.4 审批操作 API

**需求描述**：
实现审批、驳回、撤回等审批操作的 API 端点。

**功能点**：
1. 审批通过 API：`POST /api/v1/approvals/tasks/{taskId}/approve`
2. 审批驳回 API：`POST /api/v1/approvals/tasks/{taskId}/reject`
3. 撤回申请 API：`POST /api/v1/approvals/instances/{instanceId}/cancel`
4. 权限检查和状态管理

**验收标准**：
- ✅ 所有 API 端点可正常调用
- ✅ 权限检查正确
- ✅ 流程状态正确更新
- ✅ 任务状态正确更新
- ✅ 集成测试通过

---

### 2.5 表单字段 API

**需求描述**：
实现表单字段查询接口，用于前端条件构造器获取字段列表。

**功能点**：
1. 查询表单的所有字段定义
2. 返回字段的类型、选项等信息
3. 包含系统字段（提交人、提交时间等）

**验收标准**：
- ✅ API 端点可正常调用
- ✅ 返回正确的字段列表
- ✅ 包含系统字段
- ✅ 前端可正确使用

---

### 2.6 流程设计器 UI

**需求描述**：
实现完整的流程设计器前端，支持节点拖拽、连线、配置等操作。

**功能点**：
1. 画布组件：节点拖拽、连线绘制、缩放平移
2. 节点编辑器：基本信息、审批人、驳回策略、条件分支配置
3. 路由编辑器：条件表达式编辑、优先级设置
4. 节点调色板：节点类型选择
5. 状态管理：Pinia 状态管理

**验收标准**：
- ✅ 所有组件可正常使用
- ✅ 节点可正确拖拽和连线
- ✅ 配置可正确保存
- ✅ 流程可正确发布
- ✅ 端到端测试通过

---

### 2.7 查询接口

**需求描述**：
实现待办、已办、我发起的、审批时间线等查询接口。

**功能点**：
1. 待办列表：`GET /api/v1/approvals/pending`
2. 已办列表：`GET /api/v1/approvals/completed`
3. 我发起的：`GET /api/v1/approvals/initiated`
4. 审批时间线：`GET /api/v1/approvals/instances/{instanceId}/timeline`

**验收标准**：
- ✅ 所有 API 端点可正常调用
- ✅ 返回正确的数据
- ✅ 支持分页
- ✅ 前端页面可正常显示

## 四、用户故事

### 4.1 流程设计者

**故事 1**：创建条件分支流程
```
作为流程设计者
我想创建一个包含条件分支的流程
以便根据表单数据自动路由到不同的审批人

验收标准：
- 可以添加 CONDITION 节点
- 可以配置多个分支
- 每个分支可以配置条件表达式
- 可以设置默认分支
- 流程可以正确发布
```

**故事 2**：配置驳回策略
```
作为流程设计者
我想为审批节点配置驳回策略
以便控制驳回后的流程走向

验收标准：
- 可以选择 TO_START 或 TO_PREVIOUS
- 配置可以正确保存
- 驳回时按照策略处理
```

### 4.2 审批人

**故事 3**：审批表单
```
作为审批人
我想在待办列表中看到待审批的表单
并可以进行审批、驳回或撤回操作

验收标准：
- 待办列表显示所有待审批任务
- 可以查看表单详情
- 可以进行审批操作
- 可以进行驳回操作
- 可以查看审批时间线
```

**故事 4**：查看已办任务
```
作为审批人
我想查看已办任务列表
以便了解自己的审批历史

验收标准：
- 已办列表显示所有已处理任务
- 可以按时间排序
- 可以查看审批意见
```

### 4.3 表单发起人

**故事 5**：撤回申请
```
作为表单发起人
我想撤回已提交的申请
以便修改后重新提交

验收标准：
- 可以在我发起的列表中看到申请
- 可以撤回未完成的申请
- 撤回后所有待处理任务被取消
```

- 后端：Python 3.10 + FastAPI + SQLAlchemy
- 前端：Vue 3 + TypeScript + Naive UI
- 数据库：PostgreSQL
- 不能修改现有的核心数据模型结构

### 5.2 时间约束

- 第一阶段（P0）：2 周
- 第二阶段（P1）：2 周
- 总计：4-6 周

### 5.3 资源约束

- 后端开发：1-2 人
- 前端开发：1-2 人
- 测试：1 人
