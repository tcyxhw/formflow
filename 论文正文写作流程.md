【任务】
请帮我撰写毕业论文第X章的内容。

【我的论文题目】
基于FastAPI的低代码高校表单审批平台的设计与实现

【参考论文】
@论文参考

【写作流程】

1. 【结构分析】
   → 分析参考论文的**二级标题结构**
   → **判断每个标题是否可套用**：
     - 通用型标题（如"研究背景"、"目的与意义"、"国内外研究现状"）→ 可以套用
     - 领域特定型标题（如"脐橙种植管理"、"库存管理"）→ 不能套用，需根据项目重新定义

2. 【正文编写手法分析】（核心步骤）
   对参考论文每个正文段落进行**多维度分析**：

   | 分析维度 | 具体问题 |
   |:---------|:---------|
   | **论点结构** | 正文中包含几个论点？每个论点的展开方式是什么？ |
   | **内容要素** | 提到了哪些方面的内容？（如：现状描述→问题分析→意义阐述） |
   | **论述层次** | 是先讲大背景→再缩小到具体领域→最后落到本系统的意义？ |
   | **段落衔接** | 段落之间用什么转折词连接？逻辑推进是怎样的？ |

   **目的**：理解参考论文为什么这么写，而不是模仿表面形式。

3. 【格式提取】（容易被忽略的关键步骤）
   在动笔前，必须从参考论文中提取以下**精确格式参数**：

   | 格式要素 | 提取要点 |
   |:---------|:---------|
   | **标题编号与文字间距** | 章节标题中编号与标题文字之间是否有多余空格？几个？ |
   | **分点标题与正文关系** | 序号标题（如（1））是**独占一行后另起一段**描述，还是**标题描述连在一段**？ |
   | **正文换行策略** | 是一个意思一段？还是多个小点合并成一段？ |
   | **表格格式** | 参考论文中用表格呈现的内容（如软件开发平台），必须生成相同格式的markdown表格，禁止用纯文字段落替代 |

   必须将上述提取结果作为格式约束写入正文输出，不得擅自简化。

4. 【源码分析】
   → 从项目源码中自动提取技术栈（后端框架、数据库、前端框架、关键库）
   → 从源码中提取核心功能模块
   → **禁止要求用户提供以上信息**

5. 【内容编写】
   → 根据正文编写手法分析的结果，使用相同的**编写手法**展开正文
   → 将参考论文的**技术名词**替换为自己的技术名词
   → 将参考论文的**功能描述**替换为自己的功能描述
   → 段落之间承接自然，用转折词连接

   **防机械模仿约束（本节集中声明，优先级高于手法模仿）：**
   → **标题个数**：二级/三级标题个数由项目实际源码分析结果决定，不是参考论文有几个就写几个
   → **段落展开点数**：由项目实际功能决定，不是参考论文有几段就写几段
   → **篇幅对标**：生成内容控制在参考论文对应小节的 1.0-1.2 倍以内
   → **篇幅判断标准**：把该说清楚的内容说清楚即可，参照示例的详略程度
   → **参考的是编写手法**（论述层次、论点展开方式、段落衔接风格），而非字数和段落数
   → **内容深度**：由项目实际功能决定，避免为了对齐篇幅而注水或删减

6. 【超越要求】
   在完成基础模仿后，必须做到：
   → 技术选型处：加入对比分析，说明为什么选这个技术而不是其他同类技术
   → 方案设计处：说明为什么这样设计而不是那样设计，体现设计判断
   → 功能实现处：从代码中提取真实的实现细节和遇到的技术问题
   → **这些判断和细节是示例没有的，是本文独有价值**

7. 【禁止错误】
   → 【禁止】内容重复：同一观点不出现两次
   → 【禁止】衔接跳跃：论述某个领域时中途不能切换到另一领域
   → 【禁止】内容堆砌：每句话必须有存在意义
   → 【禁止】论述跑题：说着说着跑到其他领域或错误主体
   → 【禁止】虚构数据
   → 【禁止】虚构文献引用
   → 【禁止】使用第一人称
   → 【禁止】格式不一致：标题间距、分点格式与参考论文不对齐
   → 【禁止】表格格式丢失：参考论文用表格 → 必须生成markdown表格，禁止生成文字段落

8. 【写作规范】

   **格式规范（直接继承自参考论文）：**
   → **标题间距**：参考论文 `## 1.2  系统开发的目的与意义` 中编号后有两个空格，输出时必须保持一致；三级标题同理
   → **分点格式**：参考论文1.2节中"（n）标题"独占一行，"描述"另起一段；不得将标题与描述压缩到同一段落
   → **正文段落**：每个完整的意思块独占一段，段内不��行
   → 格式一旦确认，**同一文件内不得出现两种不同的分点格式**

   **Word导出兼容规范（防止正文变成"正文文本"或"first Paragraph"）：**
   → **章节标题下方紧跟的第一段正文前必须留有一个空行**（即标题行结束后换行一次再开始正文段落），否则部分markdown转docx工具会将标题下方第一段错误识别为"first Paragraph"或"正文文本"而非"正文"
   → **正文段落前后均以空行分隔**，确保每段文本被识别为独立的"正文"样式段落
   → 标题行本身与上方内容之间也应有一个空行的距离，保持层级清晰
   → 不要在标题行末尾添加额外的换行符导致行内空格异常，严格保持编号后两空格、标题文字后一换行

   段落规范：
   → 段落内部逻辑必须清晰，读者能一眼看懂
   → 绝对不重复已经说过的内容
   → 同一段落只论述一个主体，禁止混搭

   图表规范：
   → 每张图表必须有"如图X所示"或"如表X所示"
   → **同一章节内图表描述方式必须统一**：
     有文字描述则全部有，没有则全部没有

   引用规范：
   → 引用标注放在逗号或句号前
   → 引用是支撑判断的，不是替代判断的
   → 先写自己的判断，再用引用支撑

   表述规范：
   → 主语用"本系统"或"该功能"，不用第一人称
   → 不用"实际应用"类表述
   → 不用具体数据，用定性描述

9. 【技术对应】
   → 核心主题（FastAPI、低代码）必须在每章自然体现
   → 在目的与意义章节，**每个要点必须提到本系统的技术栈**
   → 技术介绍必须最终落到本系统如何使用这个技术上

10. 【篇幅控制】
     → 参考示例论文的**编写手法和结构**，而非字数和段落数
     → 核心内容简洁表述
     → **内容深度由项目实际功能决定**
     → **篇幅对标**：生成内容控制在参考论文对应小节的1.0-1.2倍以内

【输出要求】
按照上述流程完整输出本章内容。

输出前**必须确认**以下两点：
1. 每个标题的编号格式与参考论文完全一致（标题文字前有两个空格）
2. 每个分点（如（1）（2））的格式与参考论文完全一致（标题独行，描述另起一段）

最终以.md文件交付，Word样式对应关系如下：

| Markdown标记 | Word样式 |
|:------------|:--------|
| `# X 章节标题` | 标题1 |
| `## X.X  章节标题`（编号后两空格） | 标题2 |
| `### X.X.X  小节标题`（编号后两空格） | 标题3 |
| 正文段落（标题与正文之间留空一行） | 正文 |
| （n）分点标题独行，下方另起一段为描述 | 正文（每个分点描述为独立正文段落） |
| 图表说明文字 | 正文 |

**关键注意事项**：Word导出后，检查标题下方第一段是否为"正文"样式而非"first Paragraph"。如出现"first Paragraph"，说明该段之前缺少空行分隔，应在md中确保章节标题与正文之间有一个且仅有一个空行。
