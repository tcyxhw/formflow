# Vue 3 代码编写规范

## 目标
在生成、修改、重构 Vue 3 代码时，始终遵循以下原则：

- 优先使用 Vue 3 官方推荐方式
- 优先使用 TypeScript
- 优先保持组件职责单一
- 优先保证状态流清晰
- 优先与现有项目风格保持一致
- 优先最小必要改动，避免无关重构

如果本规范与现有项目稳定约定冲突：
1. 优先遵循项目现有约定
2. 如现有写法明显存在风险或反模式，先说明再调整
3. 不要为了“更现代”而做无必要大改

---

## 一、基础技术选择

默认采用：

- Vue 3
- Composition API
- `<script setup>`
- TypeScript
- Pinia（如项目使用状态管理）
- Vue Router
- 基于模块分层的 API 请求封装

除非项目已经稳定采用其他风格，否则不要退回 Options API。

---

## 二、必须遵守的硬规则

## 1. 组件编写方式
- 新组件默认使用 `<script setup lang="ts">`
- 优先使用 Composition API
- 组件必须职责单一
- 不要把页面级复杂业务全塞进一个组件
- 不要在模板中堆积复杂表达式
- 不要把大量业务逻辑直接写在 template 内

## 2. TypeScript 规则
- Props、Emits、函数参数、返回值尽量显式类型化
- 禁止无说明地滥用 `any`
- 接口数据、表单数据、状态对象尽量显式建模
- 对可空值、可选字段、异步返回值保持类型清晰

## 3. Props / Emits 规则
- Props 必须清晰、稳定、语义明确
- 不要把过多零散参数堆成超长 props 列表
- Emits 必须明确事件名称和载荷结构
- 子组件不得直接修改父组件传入的 props
- 需要双向绑定时使用约定式方式，不做隐式状态污染

## 4. 状态管理规则
- 局部状态放组件内部
- 跨组件共享状态优先放 composable 或 Pinia
- 不要把所有状态都堆到全局 store
- Store 只放真正需要跨页面/跨组件共享的数据
- Store 中避免写与视图强绑定的临时状态

## 5. API 调用规则
- 禁止在页面和组件中到处直接写裸请求
- API 请求统一封装在 `src/api` 或等价目录
- 请求参数和响应结构尽量类型化
- 错误处理、超时、鉴权、重试策略应有统一约定
- 组件中只关心“调用接口拿结果”，不要关心底层请求实现细节

## 6. 路由规则
- 页面路由按业务分组
- 路由配置清晰、命名统一
- 页面组件与可复用组件分层明确
- 路由懒加载优先用于页面级组件
- 权限、鉴权、跳转逻辑集中处理，不分散到各页面重复实现

## 7. 样式规则
- 样式作用域策略必须一致
- 避免深层嵌套和过强选择器耦合
- 不要在组件里堆大量难维护的样式逻辑
- 样式命名保持稳定、可读
- 避免无意义的内联样式

---

## 三、默认遵循的推荐规则

## 1. 组件设计
推荐把组件分为：
- 页面组件（views/pages）
- 业务组件
- 通用组件
- 布局组件

规则：
- 页面组件负责组装
- 业务组件负责业务展示与交互
- 通用组件尽量无业务耦合
- 通用组件 API 应稳定、简单、可复用

### 建议拆分标准
当组件出现以下情况时应考虑拆分：
- 文件明显过长
- 同时处理多个业务域
- 模板层级过深
- script 逻辑过重
- 同时管理多个不相关状态
- 可复用片段重复出现

## 2. 模板规范
- 模板保持简洁
- 复杂逻辑移入 `computed`、函数或 composable
- `v-if` / `v-for` 组合使用时注意可读性与性能
- 列表渲染必须使用稳定 key
- 避免在模板中调用高成本函数
- 避免在模板内写过长三元表达式

## 3. 响应式规范
- 简单基本类型优先 `ref`
- 对象/集合状态按需使用 `reactive`
- 不要混乱地嵌套解构导致丢失响应性
- 对派生状态优先使用 `computed`
- 对副作用才使用 `watch` / `watchEffect`
- 不要把本该是 `computed` 的逻辑写成 `watch`

## 4. 事件与交互
- 事件名表达业务含义
- 尽量避免多层级事件透传
- 跨层通信优先 composable、store 或清晰中介模式
- 用户交互应处理 loading、empty、error 等状态
- 提交类操作应防止重复触发

## 5. 表单处理
- 表单状态结构应清晰
- 校验逻辑集中，不要散落在各处
- 提交前进行输入合法性检查
- 错误提示文案明确
- 异步提交应处理 loading、禁用、失败提示、成功反馈

---

## 四、Composables 规范

`composables` 的目标是复用状态逻辑与副作用逻辑。

规则：
- 文件名使用 `useXxx`
- 函数名使用 `useXxx`
- 一个 composable 尽量只负责一个明确主题
- 返回值结构保持稳定
- 不要在 composable 中隐藏过多全局副作用
- 需要时显式暴露：
  - state
  - computed
  - actions
  - cleanup 能力

适合抽到 composable 的场景：
- 列表查询逻辑
- 分页逻辑
- 表单逻辑
- 异步请求状态管理
- 权限判断
- 事件监听与清理
- 复用交互逻辑

不适合抽到 composable 的场景：
- 只在一个小组件里使用一次且非常简单的逻辑
- 强耦合特定页面布局的临时逻辑

---

## 五、Pinia Store 规范

Store 应聚焦“共享业务状态”，不要成为杂物间。

规则：
- Store 名称清晰表达业务域
- state、getters、actions 语义明确
- 不要把纯展示层临时状态大量塞进 store
- 异步 action 需处理 loading / error 语义
- 不要在多个 store 间形成复杂循环依赖
- 对需要持久化的状态明确策略
- 对服务端数据缓存与本地 UI 状态分开考虑

推荐：
- 一个业务域一个 store
- action 负责流程，组件负责触发
- getter 保持纯净，不产生副作用

不推荐：
- 把所有页面状态都堆在一个超级 store
- 在 store 中直接处理 DOM
- 在 getter 中做异步逻辑

---

## 六、API 层规范

推荐目录：
- `src/api`
- `src/services`

规则：
- 请求函数命名清晰表达意图
- 入参、返回值显式类型化
- 请求实例统一管理
- 鉴权、拦截器、错误处理统一处理
- 不要在组件里重复拼接 URL、headers、错误分支

建议分层：
- `api`：与接口直接交互
- `service`：组合多个接口或做业务适配
- `component/view`：只调用 service/api，不关心底层细节

---

## 七、目录与命名规范

### 1. 文件命名
推荐：
- 组件文件：`PascalCase.vue`
- composables：`useXxx.ts`
- stores：`xxx.ts` 或 `useXxxStore.ts`
- 工具函数：按语义命名，避免模糊缩写

### 2. 组件命名
- 组件名应表达业务角色或 UI 角色
- 避免 `Test.vue`、`Demo.vue`、`Temp.vue` 这类无语义命名
- 基础组件与业务组件命名应区分层次

### 3. 变量命名
- 变量、函数使用清晰英文命名
- 布尔值尽量以 `is`、`has`、`can` 开头
- 事件处理函数使用 `handleXxx`
- 加载函数使用 `loadXxx` / `fetchXxx`

---

## 八、性能与渲染规范

- 不要在模板中放高成本表达式
- 列表渲染关注 key 的稳定性
- 大列表按需考虑虚拟滚动
- 不要对简单静态值做无意义响应式包装
- `computed` 优先于重复计算
- `watch` 只在确有副作用时使用
- 对频繁交互考虑节流/防抖
- 异步请求要避免竞态覆盖问题

---

## 九、可访问性与用户体验

- 表单控件应有明确标签
- 交互反馈应及时
- loading、empty、error 状态要可见
- 危险操作需明确确认
- 禁用态、只读态语义清晰
- 文案优先清晰而不是炫技
- 组件状态变化应尽量可预测

---

## 十、测试与可维护性

- 核心业务组件应具备可测试性
- 不要把复杂业务逻辑直接塞进模板
- 复杂逻辑优先提到 composable 或 service
- 组件应尽量通过 props、emits、store、service 明确协作
- 修 bug 时优先补对应测试或最小复现用例
- 新增逻辑时优先考虑是否会破坏现有响应式行为

---

## 十一、生成代码时的执行要求

当你生成、修改、重构 Vue 3 代码时，必须：

1. 先判断改动属于哪一层：
   - 页面层
   - 业务组件层
   - 通用组件层
   - composable
   - store
   - API/service
   - router

2. 优先保持现有项目风格一致：
   - 目录结构
   - 命名方式
   - UI 组件使用方式
   - 状态管理模式
   - 请求封装方式

3. 默认使用：
   - Vue 3
   - Composition API
   - `<script setup>`
   - TypeScript

4. 修改时优先最小改动：
   - 只改与需求直接相关部分
   - 不无故重写整个组件
   - 不无故改变 props / emits / store 接口

5. 写新代码时默认做到：
   - 类型清晰
   - 组件职责单一
   - 状态流清晰
   - API 层与视图层分离
   - loading / empty / error 状态合理

---

## 十二、禁止事项

- 禁止在模板中写过重业务逻辑
- 禁止无说明滥用 `any`
- 禁止组件直接到处写裸请求
- 禁止把所有状态都丢到全局 store
- 禁止子组件直接修改 props
- 禁止一个组件承担多个无关职责
- 禁止大量逻辑直接写在 `watch` 中
- 禁止把路由、请求、状态、DOM 操作杂糅在一起
- 禁止无意义大重构
- 禁止为了抽象而抽象

---

## 十三、输出前自检

在交付 Vue 3 代码前，检查：

- 是否使用 Vue 3 + Composition API + TypeScript
- 组件职责是否单一
- props / emits 是否清晰
- 是否把复杂逻辑移出了模板
- 是否正确选择了 ref / reactive / computed / watch
- API 调用是否有统一封装
- 是否存在滥用 store 的情况
- 是否处理 loading / empty / error
- 是否与现有项目风格一致
- 是否做了无关重构
