# Python 测试规范

## 目标
在生成、修改测试代码时，优先保证：

- 测试稳定
- 测试可读
- 测试快速
- 测试可维护
- 测试能准确描述行为而不是实现细节

如果项目中已有测试风格（例如 fixture 组织、命名方式、断言风格），优先保持一致。

---

## 一、测试框架与基本原则

默认使用：
- `pytest`

测试必须满足：
- 可重复执行
- 不依赖执行顺序
- 不依赖真实外部网络
- 不依赖真实第三方服务
- 不依赖当前机器特定环境
- 不污染全局状态
- 失败信息清晰

---

## 二、命名规范

### 1. 文件命名
优先使用以下命名方式之一：
- `test_xxx.py`
- `xxx_test.py`

### 2. 测试函数命名
测试函数名应明确表达行为与预期，例如：
- `test_create_user_returns_user_when_input_is_valid`
- `test_parse_config_raises_error_when_file_missing`

避免模糊命名：
- `test_a`
- `test_case_1`
- `test_normal`

### 3. 测试类命名
如果使用测试类：
- 类名以 `Test` 开头
- 不在测试类中写复杂共享状态
- 不要把类仅作为命名空间滥用

---

## 三、测试结构规范

推荐结构：
- Arrange：准备数据
- Act：执行行为
- Assert：断言结果

每个测试尽量只验证一个核心行为。  
如果一个测试包含多个断言，这些断言应共同服务于同一个行为验证。

优先测试：
- 输入输出
- 业务规则
- 边界条件
- 错误路径
- 回归场景

避免过度测试：
- 私有实现细节
- 重复验证第三方库行为
- 与业务价值无关的机械性断言

---

## 四、Fixture 使用规范

- 共享准备逻辑优先使用 fixture
- fixture 应小而明确，避免超级 fixture
- fixture 名称必须表达用途
- 仅在多个测试复用时抽取 fixture
- 避免 fixture 里隐藏过多副作用
- 对临时文件、临时目录优先使用 pytest 内置能力
- 对 monkeypatch 的修改必须局部且可恢复

推荐：
- 用 fixture 提供测试数据对象
- 用 fixture 提供临时依赖替身
- 用 fixture 隔离配置环境

不推荐：
- 一个 fixture 初始化整个应用所有依赖
- fixture 嵌套过深导致测试难懂
- fixture 中混杂数据库、网络、文件、环境变量修改

---

## 五、Mock / Patch 规则

仅在必要时使用 Mock。  
优先顺序：

1. 真实纯函数
2. 轻量 fake/stub
3. 局部 monkeypatch
4. mock 对象

必须 mock 的典型场景：
- 第三方 API
- 时间
- 随机数
- 文件系统副作用
- 数据库访问
- 消息队列
- 外部服务调用

规则：
- 不 mock 自己想验证的核心业务逻辑
- 不要把测试写成“验证 mock 被调用了多少次”除非这正是行为要求
- patch 位置要正确，patch 使用方而不是定义方
- Mock 只替换必要边界，不要替换整个系统

---

## 六、边界与异常测试

对以下情况必须考虑测试：

- 空输入
- 非法输入
- 缺失字段
- 极端值
- 重复数据
- 顺序异常
- 资源不存在
- 超时 / 失败 / 异常分支
- 类型不匹配
- 权限不足
- 配置缺失

异常测试要求：
- 明确断言异常类型
- 必要时断言错误信息关键片段
- 不要只写“会报错”，要写清楚为什么报错

---

## 七、数据构造规范

- 测试数据尽量最小化
- 数据尽量显式，不要过度依赖隐式默认值
- 不要在每个测试中重复堆大段样例数据
- 对复杂数据优先抽取工厂函数或 fixture
- 样例数据名称要表达业务含义

推荐：
- 小而精准的数据
- 一个测试只准备当前必要数据
- 用工厂函数生成结构化对象

不推荐：
- 无意义超长 JSON
- 多层嵌套但与断言无关的数据
- 一个测试复制另一个测试的大段数据后再手改

---

## 八、数据库与外部依赖测试

### 1. 单元测试
- 禁止依赖真实数据库
- 禁止依赖真实 HTTP 服务
- 应隔离到内存实现、fake、stub 或 mock

### 2. 集成测试
- 可以连接测试专用数据库/服务
- 必须可初始化、可清理、可重复
- 数据准备与回收必须明确
- 不允许污染开发或生产环境

### 3. 文件系统测试
- 优先使用临时目录/临时文件
- 不直接依赖固定本机路径
- 测试完成后确保资源回收

---

## 九、异步测试规则

- 异步代码使用 pytest 对应异步能力
- 正确等待协程，不遗漏 `await`
- 不要在异步测试里混入不必要的真实睡眠
- 对并发逻辑应测试：
  - 正常成功
  - 超时
  - 取消
  - 部分失败
  - 重试策略

如果使用重试、退避、轮询逻辑：
- 优先 mock 时间或等待机制
- 避免真实 `sleep` 拉长测试时间

---

## 十、断言规范

断言应尽量具体、可读、可定位问题。

推荐：
- 断言关键业务结果
- 断言结构、状态、返回值、异常类型
- 断言列表长度、字段值、状态变化

不推荐：
- 无意义断言 `assert result is not None` 代替真正业务断言
- 一个测试写十几个不相关断言
- 只断言 mock 调用次数而不验证业务结果

---

## 十一、回归测试要求

当修复 bug 时：
- 优先先写一个能复现 bug 的测试
- 再修复实现
- 最后验证测试通过

回归测试应：
- 明确描述原始缺陷
- 最小化复现路径
- 避免把修复耦合到无关逻辑

---

## 十二、覆盖策略

不是机械追求覆盖率数字，而是优先覆盖：

- 核心业务路径
- 高风险分支
- 边界输入
- 历史易出错区域
- 数据转换逻辑
- 配置解析逻辑
- 权限/状态流转逻辑

低价值覆盖包括：
- 纯样板代码
- 第三方库行为
- 与业务无关的简单包装代码

---

## 十三、生成测试代码时的执行要求

当你编写测试时，必须：

1. 先识别测试类型：
   - 单元测试
   - 集成测试
   - 回归测试
   - 异步测试

2. 优先采用最小准备数据
3. 优先隔离外部依赖
4. 测试名称必须表达行为和条件
5. 每个测试尽量只验证一个核心行为
6. 新增业务逻辑时，应同步补充相应测试
7. 修 bug 时，优先补回归测试
8. 如果现有测试风格已稳定，优先保持一致

---

## 十四、禁止事项

- 禁止依赖真实外网
- 禁止依赖生产数据库
- 禁止测试顺序耦合
- 禁止共享脏全局状态
- 禁止大量真实 `sleep`
- 禁止无意义 mock
- 禁止只测 happy path
- 禁止使用模糊命名
- 禁止把多个无关行为塞进一个测试
- 禁止把测试写成实现细节快照

---

## 十五、输出前自检

在交付测试代码前，检查：

- 测试是否稳定可重复
- 是否隔离了外部依赖
- 测试名是否表达行为
- 是否覆盖边界与失败路径
- 断言是否有业务意义
- fixture 是否过重
- 是否存在真实 sleep / 真实网络 / 固定本机路径
- 修 bug 时是否补了回归测试
- 是否与项目现有测试风格一致
