# Python 代码编写规范

## 目标
在生成、修改、重构 Python 代码时，始终遵循以下原则：

- 优先保证正确性、可读性、可维护性
- 优先与现有项目风格保持一致
- 优先最小必要改动，避免无关重构
- 明确类型、边界、异常与资源生命周期
- 对复杂逻辑保持简单、可测试、可解释

如果本规范与项目中已有稳定约定冲突：
1. 优先遵循项目现有约定
2. 如现有代码明显存在缺陷，再先说明问题再调整
3. 不要为了“更规范”而做大范围无必要改写

---

## 一、决策原则

在动手写代码前，先判断场景并选择合适范式：

### 1. IO 密集型任务
适用于：
- HTTP 请求
- 爬虫
- 文件批量读写
- 消息消费
- 数据库高并发访问

默认策略：
- 优先考虑异步方案
- 推荐 `asyncio`、`httpx`、异步数据库客户端
- 避免同步阻塞调用混入异步流程

### 2. CPU 密集型任务
适用于：
- 大量计算
- 图像处理
- 批量解析
- 数值计算
- 大规模转换

默认策略：
- 优先考虑多进程
- 避免误用线程解决 CPU 问题
- 仅在明确收益时引入并发复杂度

### 3. 大数据流/长列表处理
适用于：
- ETL
- 日志处理
- 大文件处理
- 分批传输

默认策略：
- 优先考虑生成器、迭代器、流式处理
- 禁止无必要地一次性读入超大数据
- 明确内存占用策略

### 4. 核心业务逻辑
适用于：
- 领域模型
- 业务规则
- 服务编排
- 状态转换

默认策略：
- 优先清晰建模
- 优先显式类型与边界
- 优先小函数、单一职责、清晰命名

### 5. 脚本/胶水代码
适用于：
- 临时工具
- 自动化脚本
- 运维脚本
- 一次性迁移脚本

默认策略：
- 可以更偏实用
- 但仍需保证类型、异常、日志、配置可控
- 不要因为是脚本就放弃基本工程质量

---

## 二、必须遵守的硬规则

## 1. 类型规则
- 所有公共函数、公共方法必须写完整类型注解
- 所有返回值必须显式标注类型
- 除非有充分理由，禁止无注释地使用 `Any`
- 若使用 `Any`，必须有注释说明原因
- 对结构化数据优先使用以下选择：
  - 外部输入输出边界：优先 `Pydantic`
  - 内部轻量数据结构：优先 `dataclass` / `TypedDict`
  - 简单局部对象：可用内建类型

## 2. 异常规则
- 禁止裸 `except:`
- 非必要时禁止直接 `except Exception`
- 必须捕获具体异常类型
- 不允许静默吞掉异常
- 错误处理必须满足以下任一目的：
  - 恢复流程
  - 转换为更清晰的领域错误
  - 记录日志后上抛
  - 明确降级处理
- 不要用异常代替正常流程分支

## 3. 资源管理规则
- 文件、网络连接、数据库连接、锁、流对象必须显式释放
- 优先使用 `with` 上下文管理
- 如果不能使用 `with`，必须明确说明关闭时机
- 异步资源使用异步上下文管理方式

## 4. 配置规则
- 禁止硬编码密钥、账号、URL、端口、环境差异配置
- 配置必须来自：
  - 环境变量
  - 配置文件
  - 配置对象
- 敏感信息不得写入源码
- 默认值必须合理且安全

## 5. 导入规则
- 禁止 `from xxx import *`
- 导入顺序遵循：
  1. 标准库
  2. 第三方库
  3. 本地模块
- 删除未使用导入
- 避免循环依赖
- 仅在必要时使用局部导入，并说明原因

## 6. 函数设计规则
- 单个函数应尽量保持单一职责
- 函数过长时必须拆分
- 避免隐式副作用
- 参数过多时应考虑数据对象封装
- 公共接口命名必须清晰表达意图
- 布尔参数超过一个时，应考虑重构设计

## 7. 日志规则
- 使用 `logging`，不要随意用 `print` 代替业务日志
- 错误日志必须包含足够上下文
- 不记录敏感信息
- 生产代码中避免无意义日志噪音
- 对重要边界失败、重试、降级应有明确日志

---

## 三、默认遵循的推荐规则

## 1. 风格与可读性
- 遵循 PEP 8
- 优先使用 `pathlib` 处理路径
- 优先使用 `f-string`
- 避免过度链式调用
- 避免魔法数字，必要时定义具名常量
- 复杂条件判断应拆成具名布尔变量或辅助函数
- 变量名、函数名使用 `snake_case`
- 类名使用 `PascalCase`
- 常量使用 `UPPER_SNAKE_CASE`

## 2. 文档规则
以下对象应写 docstring：
- 公共模块
- 公共类
- 公共函数
- 不直观的复杂私有函数

推荐 docstring 内容包括：
- 功能说明
- 关键参数
- 返回值
- 可能抛出的异常
- 必要时写复杂度说明

复杂度说明只在以下情况建议提供：
- 核心算法
- 性能敏感函数
- 非显而易见的复杂逻辑

不要为了形式给所有简单函数强行写复杂度标注。

## 3. 边界检查
- 入口参数应尽早做校验
- 复杂流程优先使用 Guard Clauses
- 对非法输入尽早失败
- 错误信息要明确、可定位

## 4. 可维护性
- 优先写可测试代码
- 业务逻辑与 IO 逻辑尽量分离
- 路由层/控制层保持薄
- 工具函数不要承载业务规则
- 不要过早抽象
- 相同逻辑出现两次以上再考虑抽象复用

---

## 四、场景化规则

## 1. 异步代码
- 只有在明确是 IO 密集场景时优先异步
- 异步函数内部避免阻塞式同步调用
- 避免无控制地并发创建任务
- 必要时限制并发数
- 正确处理超时、取消、重试
- 对外暴露异步接口时，异常语义要清晰

## 2. 数据处理代码
- 大文件、大列表优先流式处理
- 避免把整个输入一次性载入内存
- 批处理逻辑必须明确批大小与失败策略
- 必要时记录吞吐、耗时、失败数量

## 3. 脚本代码
- 脚本入口可以使用 `if __name__ == "__main__":`
- 业务模块不要为了形式强行加入 `__main__`
- 可执行脚本需提供清晰入口和必要参数说明

## 4. 性能敏感代码
- 先保证正确性，再做优化
- 优化前先说明瓶颈假设
- 不做无依据的“伪优化”
- 对热点逻辑，优先减少不必要对象创建、重复 IO、重复解析

---

## 五、推荐工具链

在生成或修改 Python 代码时，默认兼容以下工具链：

- 格式化：`ruff format` 或 `black`
- Lint：`ruff`
- 测试：`pytest`
- 类型检查：`mypy` 或 `pyright`

编写代码时默认目标：
- 尽量通过 lint
- 尽量便于类型检查
- 便于单元测试编写
- 不引入明显难测结构

---

## 六、生成代码时的执行要求

当你生成、修改或重构 Python 代码时，必须遵守以下工作方式：

1. 先判断当前任务属于什么场景：
   - IO 密集
   - CPU 密集
   - 数据流处理
   - 核心业务
   - 脚本/胶水代码

2. 优先保持与现有工程风格一致：
   - 目录结构
   - 命名风格
   - 类型风格
   - 日志方式
   - 测试风格

3. 修改代码时优先最小改动：
   - 只改与需求直接相关的部分
   - 不做无关重构
   - 不无端改接口、改命名、改目录

4. 写新代码时默认做到：
   - 完整类型注解
   - 清晰异常边界
   - 合理资源管理
   - 不硬编码配置
   - 具备可测试性

5. 如果需求不明确：
   - 先指出不明确点
   - 再给出保守实现
   - 不要擅自引入大框架或大改架构

---

## 七、禁止事项

- 禁止裸 `except`
- 禁止无说明的 `Any`
- 禁止 `from xxx import *`
- 禁止把敏感配置写死在代码里
- 禁止在异步代码里随意混用阻塞 IO
- 禁止把路由层写成大业务层
- 禁止为了“高级感”引入不必要的抽象
- 禁止无测试地交付核心业务逻辑
- 禁止为简单问题设计过重的数据模型
- 禁止无理由修改已有稳定接口

---

## 八、输出前自检

在交付 Python 代码前，检查：

- 范式是否合适
- 类型是否完整
- 命名是否清晰
- 是否存在裸异常处理
- 是否正确管理资源
- 是否存在硬编码配置
- 是否容易测试
- 是否与现有项目风格一致
- 是否做了无关重构
- 核心逻辑是否需要补测试

如果发现规范与现有代码冲突，优先最小改动并解释原因。
