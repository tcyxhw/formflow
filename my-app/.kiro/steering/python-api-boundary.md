# Python API 与边界层规范

## 目标
本规范只针对“边界层”代码，包括但不限于：

- HTTP API
- RPC 接口
- 消息消费入口
- 配置加载
- 数据序列化/反序列化
- DTO / Schema / ViewModel
- 与前端、外部系统、数据库交界处的数据转换

边界层的核心原则：

- 输入必须校验
- 输出必须明确
- 错误必须可控
- 配置必须外置
- 控制层必须薄
- 数据契约必须清晰

---

## 一、数据契约规则

### 1. 外部边界优先使用显式数据模型
以下场景优先使用 `Pydantic` 或等价显式 schema：
- API 请求体
- API 响应体
- 配置对象
- 消息队列载荷
- 反序列化后的外部数据
- 跨服务通信对象

禁止做法：
- 入口层直接接受无结构 `dict` 后层层透传
- 输出层直接返回未经约束的任意对象
- 依赖“大家都知道有哪些字段”的隐式约定

### 2. 内外边界分层
建议区分：
- Request DTO
- Response DTO
- Domain Model
- Persistence Model

不要把同一个模型同时拿来承担：
- API 输入
- 数据库存储
- 内部业务实体
- 前端展示对象

如果项目规模较小，可适度简化，但仍要保持边界语义清晰。

### 3. 字段定义要求
- 字段名称表达业务含义
- 必要时提供默认值、约束、说明
- 对可空字段明确声明
- 对枚举/状态值优先使用显式类型而不是魔法字符串
- 对时间、金额、ID 等关键字段保持统一表示方式

---

## 二、控制层/路由层规则

### 1. 控制层必须薄
控制层只负责：
- 接收输入
- 调用校验/解析
- 调用 service/use case
- 映射返回结果
- 处理边界错误
- 组织 HTTP/RPC 响应

控制层不负责：
- 大段业务逻辑
- 复杂数据转换
- 直接拼接 SQL
- 深层规则判断
- 跨多个系统的大规模编排

### 2. 路由处理要求
- 单个 handler 保持简洁
- 请求参数解析明确
- 返回值结构稳定
- 状态码/错误码语义清晰
- 不把框架对象向业务层深传

### 3. 依赖注入
- 显式传入依赖，避免隐藏全局依赖
- 测试时可轻松替换依赖
- 不要在路由函数中临时创建复杂基础设施对象

---

## 三、输入校验规则

### 1. 输入校验必须前置
对以下内容尽早校验：
- 必填字段
- 类型
- 长度
- 范围
- 格式
- 枚举值
- 业务前置条件

### 2. 错误信息要求
- 清晰说明哪个字段错了
- 说明期望格式或约束
- 不暴露敏感实现细节
- 面向调用方可理解

### 3. 不能信任外部输入
任何来自以下来源的数据都视为不可信：
- 前端请求
- 外部 API
- 配置文件
- 数据库原始返回
- MQ 消息
- 文件输入

必须先校验/规范化，再进入业务层。

---

## 四、输出规范

### 1. 输出必须稳定
- 返回结构应可预测
- 字段命名保持一致
- 成功与失败响应格式应明确
- 不要返回杂乱、临时拼凑的结构

### 2. 输出必须可序列化
- 返回对象应明确可 JSON 化或可协议编码
- 不要把框架内部对象、连接对象、复杂不可序列化实例直接输出

### 3. 不泄露敏感信息
禁止把以下信息直接返回给客户端：
- 堆栈详情
- 内部 SQL
- 密钥/token
- 内部路径
- 内部服务地址
- 调试敏感上下文

---

## 五、错误处理与错误映射

### 1. 错误分层
建议区分：
- 输入错误 / 校验错误
- 认证错误
- 权限错误
- 资源不存在
- 状态冲突
- 外部依赖失败
- 系统内部错误

### 2. 错误映射规则
边界层负责把内部错误映射为：
- 合适的 HTTP 状态码 / RPC 错误码
- 稳定的错误结构
- 用户可理解但不过度暴露内部细节的信息

### 3. 日志与对外响应分离
- 日志可以更详细
- 对外错误响应应更克制
- 不要把原始异常直接原样暴露给外部

---

## 六、配置规范

### 1. 配置必须集中管理
配置来源应明确，例如：
- 环境变量
- `.env`
- 配置类
- 配置文件

### 2. 配置对象建议显式建模
对于以下配置优先使用显式 schema：
- 数据库连接
- Redis / MQ
- 外部 API 凭据
- 服务开关
- 超时、重试、限流参数

### 3. 密钥管理
- 禁止把密钥写死在代码中
- `.env.example` 只保留占位，不放真实值
- 配置缺失时应尽早报错，不要延迟到深层流程才失败

---

## 七、序列化与转换规则

### 1. 显式转换
- 边界层负责 DTO 与领域对象之间的转换
- 避免让业务层依赖 HTTP/框架特定结构
- 不要在多层之间随意传裸 `dict`

### 2. 时间与金额
- 时间字段统一格式和时区策略
- 金额字段统一精度与类型策略
- ID 字段风格保持一致

### 3. 分页、列表与批量接口
- 分页参数命名保持一致
- 返回列表时，明确数据项与元信息结构
- 批量接口需说明部分成功/失败策略

---

## 八、服务层协作规则

边界层调用 service/use case 时应遵循：
- 输入对象清晰
- 输出对象清晰
- 不把框架 request/session/context 到处透传
- 不让 service 直接依赖 HTTP 细节
- 不在 service 中拼接响应对象

目标是：
- service 可被单元测试
- service 可被复用
- web 框架可替换时影响最小

---

## 九、文档与可发现性

对公共 API/边界接口，推荐清晰描述：
- 输入参数
- 返回结构
- 错误语义
- 权限要求
- 幂等性
- 限流或分页规则

新增接口时，优先保持命名与现有接口风格一致。

---

## 十、生成代码时的执行要求

当你生成 API、Schema、DTO、配置或边界层代码时，必须：

1. 先明确这是哪一类边界：
   - HTTP API
   - 配置对象
   - 外部服务适配
   - 消息载荷
   - DTO / Schema

2. 对外部输入做显式校验
3. 对输出结构做显式定义
4. 控制层保持薄，不承载核心业务
5. 不直接透传无结构数据到业务层
6. 不暴露内部异常实现细节
7. 配置必须外置
8. 修改时优先保持已有接口兼容，除非需求明确要求变更

---

## 十一、禁止事项

- 禁止在路由层写大段业务逻辑
- 禁止跨层传递杂乱裸 `dict` 作为长期契约
- 禁止把数据库模型直接当 API 响应模型长期使用
- 禁止把密钥写进源码
- 禁止把内部异常直接原样返回客户端
- 禁止响应结构反复变化且无说明
- 禁止输入不校验就进入业务层
- 禁止控制层深度依赖框架对象并向下扩散
- 禁止把 DTO、领域对象、持久化对象完全混为一体

---

## 十二、输出前自检

在交付边界层代码前，检查：

- 输入是否被显式校验
- 输出是否有清晰结构
- 错误是否被合理映射
- 是否泄露敏感信息
- 控制层是否足够薄
- 配置是否外置
- 是否把无结构数据层层透传
- 是否与现有接口风格一致
- 是否无故破坏接口兼容性
